## 阶段 8.4 第一轮压测结果(2 实例 × 10 对用户 × 1 msg/s × 30s)

### 压测拓扑
- 双实例:8080 + 8081
- 流量模式:对聊,每对一个用户在 8080,一个在 8081(全跨实例转发)
- 总流量:592 条消息 / 30 秒 ≈ 20 msg/s

### 端到端延迟(客户端 sender → 客户端 receiver)
- P50: 37.91 ms
- P95: 47.31 ms
- P99: 89.10 ms

### 服务端处理延迟分段(从日志聚合)
- create_msg (MySQL INSERT): 13-17 ms (~41% 总耗时,**最大瓶颈**)
- update_self_read (MySQL UPDATE): 6-9 ms
- mq_publish: 2-12 ms (突发下累积,**串行 Confirm 瓶颈**)
- search_user (Redis): 0.8-4 ms (WSL 环境网络抖动)
- 服务端总处理: 27-41 ms

### 关键发现
1. MySQL INSERT (create_msg) 是首要优化目标
2. 同步 Publisher Confirm 在突发流量下出现累积排队(2 → 12 ms)
3. 端到端 vs 服务端差值约 48ms 包含 WSL 网络栈开销
4. 0 丢包,跨实例转发链路稳定


### 阶段 8.4 路径对比实验 (A1 vs A2)

| 维度 | 跨实例 (A1) | 全本地 (A2) | Δ |
|---|---|---|---|
| P50 | 37.91 ms | 40.09 ms | +2 ms (噪声) |
| P95 | 47.31 ms | 53.40 ms | +6 ms (噪声) |
| **P99** | **89.10 ms** | **58.11 ms** | **跨实例 +31 ms** |
| 吞吐 | 20 msg/s | 20 msg/s | - |
| 丢包 | 0% | 0% | - |

**关键观察**:跨实例转发(Redis Pub/Sub)在 P99 长尾上比本地投递多 ~30 ms 开销,
符合 Pub/Sub "尽力而为"链路的预期。中位延迟差异在 WSL 网络抖动范围内,
未在低流量下显著体现 Pub/Sub 链路开销。


### 阶段 8.4 B1 压测结果(2 实例,100 用户,500 QPS,30 秒)

| 维度 | 数值 |
|---|---|
| 总吞吐 | 14969 条 / 30 秒 = 500 msg/s 稳定 |
| 端到端 P99 | 174 ms |
| 端到端 P95 | 148 ms |
| 端到端 P50 | 94 ms |
| 丢包率 | 0% |
| 错误率 | 0% |

**服务端处理瓶颈分析**:
- `mq_publish` 段从 A1 的 2-12ms 涨至 25-50ms,**串行 Publisher Confirm 在 250 QPS/实例 下出现明显排队累积**
- `create_msg` (MySQL INSERT) 从 15ms 涨至 32ms,**MySQL 内部锁竞争开始显现**
- 投递路径耗时(0.05ms 本地 / 1.5ms 跨实例)在总延迟中占比 < 5%,不构成瓶颈

**容量评估**:
- 单实例 100 goroutine 并发处理 ~ 1000 QPS 理论上限
- 实测 500 QPS / 双实例 仍有 ~1 倍 headroom
- 预计总 1000+ QPS 撞到拐点(待 B2 验证)
